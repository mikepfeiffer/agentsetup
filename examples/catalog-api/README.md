# Example: `catalog-api` (ASP.NET Core + EF Core)

A small **ASP.NET Core 8** minimal API — a product catalog backed by **EF Core
(SQLite)** — that is a self-contained reference for both core ideas in one folder:

1. **Shared agent instructions** — its own [`AGENTS.md`](AGENTS.md) (source of
   truth) + [`CLAUDE.md`](CLAUDE.md) (one-line import stub), layered on top of the
   repo-root instructions it inherits.
2. **Deterministic hooks** — its own [`.agent-hooks/`](.agent-hooks/) policy plus
   the thin [`.claude/settings.json`](.claude/settings.json) and
   [`.codex/hooks.json`](.codex/hooks.json) adapters.

The contrast with the [`payments-app`](../payments-app/) example is the point:
the *patterns* are identical, but the .NET-specific **content** is different.

## Launch the agent *from this directory*

```bash
cd examples/catalog-api
claude      # or: codex
```

Agent **settings/hooks load from the directory you launch in** and are *not*
inherited from parent folders — so this example's hooks fire only when this
folder is the launch root. The **root** `AGENTS.md`/`CLAUDE.md` still load via
upward inheritance, so you get root guidance + this app's stricter rules + this
app's guardrails. (See the root [`README.md`](../../README.md) §5.)

## What it does

```bash
dotnet test CatalogApi.sln                 # build + run xUnit tests (uses in-memory SQLite)
dotnet run --project src/CatalogApi        # applies migrations, seeds, serves on http://localhost:5xxx
curl localhost:5xxx/products               # -> seeded products as JSON
curl -X POST localhost:5xxx/products \
  -H 'content-type: application/json' \
  -d '{"name":"Sprocket","price":4.50}'    # -> 201 Created
```

- [`src/CatalogApi/Program.cs`](src/CatalogApi/Program.cs) — minimal API endpoints (`GET/POST /products`).
- [`src/CatalogApi/Models/`](src/CatalogApi/Models/) — `Product` (decimal `Price`) + edge validation.
- [`src/CatalogApi/Data/`](src/CatalogApi/Data/) — `CatalogDb` context, seed, design-time factory.
- [`src/CatalogApi/Migrations/`](src/CatalogApi/Migrations/) — **generated** by `dotnet ef`; don't hand-edit.
- [`tests/CatalogApi.Tests/`](tests/CatalogApi.Tests/) — `WebApplicationFactory` integration + unit tests.

## What the hooks guard (the .NET-specific bits)

| You try to… | Hook result | Why |
|---|---|---|
| edit `Migrations/*ModelSnapshot.cs` or `*.Designer.cs` | **blocked** | generated by `dotnet ef` — add a new migration instead |
| edit `appsettings.Production.json`, `*.pfx`, `secrets.json` | **blocked** | secrets/config |
| write into `bin/` or `obj/` | **blocked** | build output |
| run `dotnet ef database drop`, `dotnet nuget push`, `git push --force` | **blocked** | dangerous commands |
| edit `src/CatalogApi/Program.cs`, run `dotnet test` | allowed | ordinary work |

Try the migration guard directly (this is what the hook host pipes to stdin):

```bash
printf '{"tool_name":"Edit","tool_input":{"file_path":"src/CatalogApi/Migrations/CatalogDbModelSnapshot.cs"}}' \
  | node .agent-hooks/agent-hook.mjs pre-tool-policy; echo "exit=$?"   # -> exit=2 (blocked)
```

Run the hook tests with `npm run test:hooks`. These guardrails **supplement** —
they don't replace — code review, CI, secret scanning, and branch protection, and
they run with your local permissions. See
[`.agent-hooks/README.md`](.agent-hooks/README.md) for the full contract.

## Notes

- **EF Core pinned to 8.0.x.** The default EF packages now target .NET 10; this
  example targets the installed **.NET 8** SDK, so the packages and the
  `dotnet-ef` tool are pinned to `8.0.*` (`.config/dotnet-tools.json`). Run
  `dotnet tool restore` before `dotnet ef`.
- **`decimal` on SQLite.** SQLite has no decimal type, so EF stores `Price` as
  text. `decimal` remains the correct money type in .NET — exact, no float error.
- **Generated/ignored:** `bin/`, `obj/`, and `catalog.db` are git-ignored (and
  hook-blocked), so an agent won't "fix" generated output instead of the source.
